## API Endpoints (Phase A)

### Agents
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/agents` | Create agent |
| GET | `/api/agents` | List all agents |
| GET | `/api/agents/{id}` | Get single agent |
| PUT | `/api/agents/{id}` | Update agent |
| DELETE | `/api/agents/{id}` | Delete agent |
| POST | `/api/agents/{id}/run` | Start a run for this agent |
| GET | `/api/agents/{id}/runs` | List runs for this agent |

### Runs
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/runs` | List all runs (optional `?status=` filter) |
| GET | `/api/runs/{id}` | Get single run |
| POST | `/api/runs/{id}/cancel` | Cancel a running run |
| POST | `/api/runs/{id}/approve` | Approve an approval gate |

### WebSocket
| Path | Description |
|------|-------------|
| `/api/ws/runs/{run_id}` | Live run streaming (events below) |

WebSocket events (server -> client):
- `run_started` -- run began executing
- `agent_started` -- an agent step started
- `agent_completed` -- an agent step finished
- `screenshot` -- computer use screenshot (base64 data)
- `approval_required` -- run paused at approval gate
- `run_completed` -- run finished successfully
- `run_failed` -- run failed with error

WebSocket messages (client -> server):
- `{"type": "approval_response", "data": {"action": "approve"}}` -- approve a gate

### Health
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/health` | Health check |

## Data Models

### Agent
```typescript
interface Agent {
  id: string;
  name: string;
  description: string;
  type: "agent" | "approval" | "input" | "output";
  status: "creating" | "ready" | "error";
  forge_path: string;
  samples: string[];
  input_schema: SchemaField[];
  output_schema: SchemaField[];
  computer_use: boolean;
  forge_config: Record<string, any>;
  provider: string;   // "anthropic", "claude_code", "openai"
  model: string;      // "claude-sonnet-4-6", "gpt-4o", etc.
  created_at: string; // ISO datetime
  updated_at: string;
}

interface SchemaField {
  name: string;
  type: string;  // "text", "number", "boolean", etc.
  required: boolean;
}
```

### AgentCreate (POST body)
```typescript
interface AgentCreate {
  name: string;          // 1-200 chars
  description?: string;  // max 10000 chars
  samples?: string[];
  computer_use?: boolean; // default false
  provider?: string;      // default "anthropic"
  model?: string;         // default "claude-sonnet-4-6"
}
```

### Run
```typescript
interface Run {
  id: string;
  project_id: string | null;  // null for standalone agent runs
  agent_id: string | null;
  status: "queued" | "running" | "awaiting_approval" | "completed" | "failed";
  inputs: Record<string, any>;
  outputs: Record<string, any>;
  started_at: string | null;
  completed_at: string | null;
}
```

### AgentRun (step within a run)
```typescript
interface AgentRun {
  id: string;
  run_id: string;
  node_id: string;
  status: "pending" | "running" | "completed" | "failed" | "skipped";
  inputs: Record<string, any>;
  outputs: Record<string, any>;
  logs: string;
  duration_ms: number;
  started_at: string | null;
  completed_at: string | null;
}
```

## Key UX Flows

### 1. Create Agent
Dashboard -> "New Agent" button -> Agent Editor form -> Fill fields -> Submit POST /api/agents -> Agent appears in list with status "creating" -> WebSocket updates status to "ready" when forge generation completes

### 2. Run Agent
Agent detail/list -> "Run" button -> Optional: fill inputs based on agent.input_schema -> POST /api/agents/{id}/run -> Redirect to Run Viewer -> WebSocket streams execution events

### 3. View Run
Runs list or redirect after starting -> Run Viewer page -> Shows execution steps timeline -> Live log streaming via WebSocket -> Final outputs displayed when run_completed

### 4. Cancel Run
Run Viewer -> "Cancel" button -> POST /api/runs/{id}/cancel -> Status updates to "failed"

## Visual Quality Gate (MANDATORY)

Aesthetics matter. The UI must not look like a hackathon project. After building each page, use computer use to screenshot it and critically evaluate:

- **Spacing**: Consistent padding/margins. Nothing cramped, nothing floating in empty space.
- **Alignment**: Elements are aligned on a grid. Labels line up with inputs. Cards are the same height.
- **Borders**: Subtle, consistent. No thick ugly borders. Use `border-[#1e293b]` or similar from the theme.
- **Typography**: Clear hierarchy. Headings are visually distinct from body text. No font-size soup.
- **Colors**: Dark theme is cohesive. Green accent is used sparingly (buttons, active states, badges) -- not plastered everywhere.
- **Hover/focus states**: Buttons and interactive elements have visible hover/focus feedback.
- **Empty states**: Pages with no data show a helpful empty state, not a blank void.
- **Responsiveness**: Content fills the available space naturally. No tiny centered cards on a wide screen.
- **Cards and containers**: Rounded corners, subtle shadows, consistent treatment across all pages.
- **Tables**: Clean rows, alternating backgrounds or clear separators, readable on dark backgrounds.

Compare every page against the wireframes in `api/docs/wireframe-*.png`. If something looks off -- borders too thick, spacing uneven, text hard to read, layout feels amateur -- fix it before moving on. Take pride in the craft.

This is a blocking gate. Do not commit until every page looks polished.
