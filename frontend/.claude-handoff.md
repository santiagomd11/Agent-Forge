## API Endpoints (Phase A)

### Agents
| Method | Path | Description |
|--------|------|-------------|
| POST | `/api/agents` | Create agent |
| GET | `/api/agents` | List all agents |
| GET | `/api/agents/{id}` | Get single agent |
| PUT | `/api/agents/{id}` | Update agent |
| DELETE | `/api/agents/{id}` | Delete agent |
| POST | `/api/agents/{id}/run` | Start a run for this agent |
| GET | `/api/agents/{id}/runs` | List runs for this agent |

### Runs
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/runs` | List all runs (optional `?status=` filter) |
| GET | `/api/runs/{id}` | Get single run |
| POST | `/api/runs/{id}/cancel` | Cancel a running run |
| POST | `/api/runs/{id}/approve` | Approve an approval gate |

### WebSocket
| Path | Description |
|------|-------------|
| `/api/ws/runs/{run_id}` | Live run streaming (events below) |

WebSocket events (server -> client):
- `run_started` -- run began executing
- `agent_started` -- an agent step started
- `agent_completed` -- an agent step finished
- `screenshot` -- computer use screenshot (base64 data)
- `approval_required` -- run paused at approval gate
- `run_completed` -- run finished successfully
- `run_failed` -- run failed with error

WebSocket messages (client -> server):
- `{"type": "approval_response", "data": {"action": "approve"}}` -- approve a gate

### Health
| Method | Path | Description |
|--------|------|-------------|
| GET | `/api/health` | Health check |

## Data Models

### Agent
```typescript
interface Agent {
  id: string;
  name: string;
  description: string;
  type: "agent" | "approval" | "input" | "output";
  status: "creating" | "updating" | "ready" | "error";
  forge_path: string;
  steps: string[];
  samples: string[];
  input_schema: SchemaField[];   // auto-generated by forge, read-only on create
  output_schema: SchemaField[];  // auto-generated by forge, read-only on create
  computer_use: boolean;
  forge_config: Record<string, any>;
  provider: string;   // "anthropic", "claude_code", "openai"
  model: string;      // "claude-sonnet-4-6", "gpt-4o", etc.
  created_at: string; // ISO datetime
  updated_at: string;
}

interface SchemaField {
  name: string;
  type: string;  // "text", "number", "boolean", etc.
  required: boolean;
}
```

### AgentCreate (POST body)
```typescript
interface AgentCreate {
  name: string;          // 1-200 chars
  description?: string;  // max 10000 chars
  steps?: string[];       // workflow steps (e.g., ["Research", "Outline", "Write Draft"])
  samples?: string[];     // quality sample texts for calibration
  computer_use?: boolean; // default false
  provider?: string;      // default "anthropic"
  model?: string;         // default "claude-sonnet-4-6"
}

// Note: input_schema and output_schema are NOT set on create.
// They are auto-generated by forge during agent generation.
// On update (PUT), they CAN be set manually if needed.
```

## Key UX Flows

### 1. Create Agent
Dashboard -> "New Agent" button -> Agent Editor form -> Fill fields -> Submit POST /api/agents -> Agent appears in list with status "creating" -> WebSocket updates status to "ready" when forge generation completes

### 2. Run Agent
Agent detail/list -> "Run" button -> Optional: fill inputs based on agent.input_schema -> POST /api/agents/{id}/run -> Redirect to Run Viewer -> WebSocket streams execution events

### 3. View Run
Runs list or redirect after starting -> Run Viewer page -> Shows execution steps timeline -> Live log streaming via WebSocket -> Final outputs displayed when run_completed

### 4. Cancel Run
Run Viewer -> "Cancel" button -> POST /api/runs/{id}/cancel -> Status updates to "failed"

## Error Handling

API errors follow this envelope:
```json
{
  "error": {
    "code": "AGENT_NOT_FOUND",
    "message": "Agent with id 'xxx' not found",
    "details": {}
  }
}
```

Error codes: `AGENT_NOT_FOUND`, `RUN_NOT_FOUND`, `PROJECT_NOT_FOUND`, `RUN_NOT_ACTIVE`, `NO_GATE_PENDING`, `AGENT_BUSY`, `AGENT_NOT_READY`

## Agent Status Lifecycle

- `creating` -- initial status, forge is generating the agent files
- `updating` -- a substantive field was changed, forge is re-generating
- `ready` -- agent is ready to run
- `error` -- forge generation failed

**Substantive fields** (trigger re-generation): `description`, `steps`, `samples`, `computer_use`
**Cosmetic fields** (no re-generation): `name`, `provider`, `model`, `input_schema`, `output_schema`

When an agent is `creating` or `updating`, the API returns 409 AGENT_BUSY if you try to change substantive fields.
When an agent is not `ready`, the API returns 409 AGENT_NOT_READY if you try to run it.

The UI should:
- Show a spinner/loading state for agents in `creating` or `updating` status
- Disable the "Run" button when status is not `ready`
- Disable substantive field editing when status is `creating` or `updating`
- Poll or use WebSocket to detect when status changes to `ready`
